# VPinPlay Table Metadata Sync Service — Design Specification

## 1. Overview

This document defines the design for a VPinPlay metadata sync service.

The service allows VPinPlay clients and other approved programs to submit table metadata through a Python REST API and persist the data in a MongoDB NoSQL backend.

The system is designed to:

- accept metadata submissions from client programs
- separate global table metadata from user-specific metadata
- use `Info.VPSId` as the canonical table key
- store current state only for normal reads
- optionally track changes over time
- support full-state syncs from clients on startup or at intervals

This specification covers:

- architecture
- data model
- sync rules
- JSON payload format
- REST API endpoints
- merge and update rules
- indexing guidance
- future extension points

---

## 2. Technology Choices

### Backend database

- **MongoDB**
- document-oriented NoSQL database
- flexible schema for evolving metadata
- good fit for per-table and per-user state documents

### API layer

- **Python**
- recommended framework: **FastAPI**
- Pydantic models for request validation
- simple JSON REST endpoints for submissions and queries

### Why this stack

MongoDB is a good fit because the metadata schema may evolve over time and the data is naturally document-shaped. FastAPI is a good fit because it makes it easy to validate incoming JSON blobs, build clean REST endpoints, and generate OpenAPI documentation.

---

## 3. Core Design Principles

1. **`Info.VPSId` is always the primary table identity key.**
2. **Global metadata and user-specific metadata are stored separately.**
3. **Clients may send full current state on startup.**
4. **The backend computes differences.**
5. **Normal reads return only the latest current state.**
6. **History and change tracking are optional but supported.**
7. **`VPinPlay.altvpsid` is stored as plain data only and does not affect identity.**

---

## 4. Data Classification

### 4.1 Global table metadata

These fields are considered shared table metadata and belong in the canonical `tables` collection.

From `Info`:

- `VPSId`
- `Rom`

From `VPXFile`:

- all fields

### 4.2 User-specific metadata

These fields are per-user or per-cabinet state and belong in `user_table_state`.

From `User`:

- `Rating`
- `LastRun`
- `StartCount`
- `RunTime`
- `Score`

From `VPinPlay`:

- `altvpsid`

### 4.3 Ignored by this service

These sections are currently not submitted or not stored by the sync service:

- all other `VPinPlay` fields except `altvpsid`
- `Medias`
- `Info` fields other than `VPSId` and `Rom`

These may remain local to VPinPlay or be handled by a different service later.

---

## 5. Canonical Identity Rule

The table identity is based on:

```text
Info.VPSId + VPXFile (exact match)
```

Multiple variations of the same `VPSId` can exist if they have different `VPXFile` data.

This allows storing different versions or configurations of the same table.

- Each unique combination creates a separate document in the `tables` collection
- `VPSId` is used as a query key to retrieve all variations
- The document `_id` is auto-generated by MongoDB

---

## 6. Collections

### 6.1 Tables collection

- **Purpose**: Store global table metadata variations
- **Document structure**:
  - `_id`: Auto-generated MongoDB ObjectId
  - `vpsId`: String (from Info.VPSId)
  - `rom`: String (from Info.Rom)
  - `vpxFile`: Object (from VPXFile section)
  - `updatedAt`: DateTime
  - `lastSeenAt`: DateTime
- **Key behavior**: Multiple documents can exist with the same `vpsId` if `vpxFile` differs
- **Indexes**: `vpsId` for efficient querying

### 6.2 User table state collection

- **Purpose**: Store per-user table state
- **Document structure**:
  - `_id`: Auto-generated
  - `userId`: String
  - `vpsId`: String
  - `rating`: Integer
  - `lastRun`: DateTime or Unix timestamp
  - `startCount`: Integer
  - `runTime`: Integer
  - `score`: Object or null
  - `altvpsid`: String
  - `updatedAt`: DateTime
  - `lastSeenAt`: DateTime
- **Key**: Compound (`userId`, `vpsId`)
- **Indexes**: Compound index on (`userId`, `vpsId`)

### 6.3 Client registry collection

- **Purpose**: Track userId-machineId pairings
- **Document structure**:
  - `_id`: Auto-generated
  - `userId`: String (unique)
  - `initials`: String
  - `machineId`: String (unique, 64 chars)
  - `registeredAt`: DateTime
- **Indexes**: `userId` (unique), `machineId`

## 7. Sync Model

### 7.1 Client behavior

A client program may send a **full current-state snapshot** on startup.

This means the client sends all tables it currently knows about, using the defined JSON format.

### 7.2 Backend behavior

The backend does **not** treat every sync as entirely new data.

Instead, for each submitted table:

1. locate the canonical table by `VPSId`
2. compare incoming global fields with the existing `tables` document
3. compare incoming user fields with the existing `user_table_state` document
4. update only changed fields
5. optionally record change events
6. update `lastSeenAt`

### 7.3 If nothing changed

If the submitted data matches the stored current state:

- no change event is created
- no meaningful update is applied
- `lastSeenAt` may be updated

### 7.4 If a single field changed

Example: only `Rating` changed.

Then:

- only that user field is updated
- only that field appears in the optional change log
- the current documents remain the source of truth for normal reads

---

## 8. Merge Rules

### 8.1 Global fields

Global fields come from:

- `Info.VPSId`
- `Info.Rom`
- all `VPXFile` fields

Recommended merge rules:

- `vpsId`: immutable once set because it is the key
- `rom`: set only on first submission, never updated
- `vpxFile.*`: set only on first submission, never updated
- `lastSeenAt`: always update on successful sync
- `updatedAt`: set only on first submission

### 8.2 User-specific fields

Recommended merge rules:

- `rating`: replace with latest submitted value
- `lastRun`: keep the newest non-null timestamp
- `startCount`: keep the maximum value
- `runTime`: keep the maximum value
- `score`: replace with latest submitted object-like value
- `altvpsid`: replace with latest submitted value
- `lastSeenAt`: always update on successful sync
- `updatedAt`: update only if one or more user fields changed

## 10. REST API Design

Base path:

```text
/api/v1
```

### 10.1 Submit full sync

**POST** `/api/v1/sync`

Accepts a full-state snapshot from a client program.

#### Request body

Uses the full sync JSON format defined above.

#### Response example

```json
{
  "status": "ok",
  "receivedAt": "2026-03-23T15:00:01Z",
  "summary": {
    "tablesReceived": 250,
    "tablesCreated": 2,
    "tablesUpdated": 4,
    "userStatesCreated": 3,
    "userStatesUpdated": 6,
    "unchanged": 235,
    "errors": 0
  }
}
```

### 10.2 Get table variations by VPSID

**GET** `/api/v1/tables/{vpsId}`

Returns all unique variations of the table with the given VPSID.

#### Response example

```json
{
  "vpsId": "lkSumsrF",
  "rom": "ij_l7",
  "vpxFile": {
    "filename": "Indiana Jones The Pinball Adventure (Williams 1993).vpx",
    "filehash": "5b5d9f2c4a44116f1575f088334156828d023ef604b1ad9673774bd31da7be1b",
    "version": "1.1"
  },
  "updatedAt": "2026-03-23T15:00:00Z",
  "lastSeenAt": "2026-03-23T15:00:00Z"
}
```

### 10.3 Get user state for a table

**GET** `/api/v1/users/{userId}/tables/{vpsId}`

Returns the current per-user state for a table.

#### Response example

```json
{
  "userId": "cabinet_1",
  "vpsId": "lkSumsrF",
  "rating": 5,
  "lastRun": null,
  "startCount": 0,
  "runTime": 0,

    "rom": "ij_l7",

  "lastSeenAt": "2026-03-23T15:00:00Z"
}
```

### 10.4 Check if userId is available

**GET** `/api/v1/users/{userId}/available`

Returns whether a userId is available for registration.

#### Response example

```json

  "available": true

```

**GET** `/api/v1/users/{userId}/initials`

## 11. Sync Processing Algorithm

For each submitted table in `/sync`:

### 11.1 Validate

- ensure `info.vpsId` is present
- validate value types
- normalize missing optional fields to defaults if needed

### 11.2 Upsert canonical table

Lookup:

```text
tables._id = info.vpsId
```

If not found:

- create new `tables` document

If found:

- compare `rom`
- compare all `vpxFile.*` fields
- update only changed fields

### 11.3 Upsert user state

Lookup:

```text
user_table_state.userId = client.userId
user_table_state.vpsId = info.vpsId
```

If not found:

- create new `user_table_state` document

If found:

- compare `rating`
- compare `lastRun`
- compare `startCount`
- compare `runTime`
- compare `score`
- compare `altvpsid`
- apply merge rules

### 11.4 Record optional change events

If enabled, write change events only when one or more meaningful fields changed.

### 11.5 Return summary

Return aggregate counts of:

- created tables
- updated tables
- created user states
- updated user states
- unchanged records
- validation errors

## 13. Validation Rules

Recommended validation rules:

- `vpsId` must be non-empty string
- `rom` should be string or null
- `rating` should be integer within an allowed range, such as `0` to `5`
- `startCount` should be integer `>= 0`
- `runTime` should be integer `>= 0`
- `lastRun` should be Unix timestamp (integer) or ISO timestamp string or null
- `score` should be an object or null; array and scalar score payloads should be rejected or normalized
- boolean detection fields should be booleans
- if legacy clients send strings like `"true"` and `"false"`, the API should normalize them during validation

Client validation rules:

- `userId`, `initials`, and `machineId` are required in the client section
- First submission with a `userId` registers it with the provided `machineId`
- Subsequent submissions must use the same `machineId` for that `userId`
- `initials` is stored with that `userId` + `machineId` pairing and may be updated on later successful syncs
- Invalid submissions (wrong machineId for existing userId) are rejected

## 15. Example Read Patterns

### 15.1 Read current canonical table

Query `tables` by `vpsId`.

### 15.2 Read current user state

Query `user_table_state` by `(userId, vpsId)`.

### 15.3 Read merged current view for UI

Join in application code:

- `tables` document
- matching `user_table_state` document

This produces a combined view for frontend display.

### 15.4 Read only changed data

If `table_changes` is enabled, query by `since` timestamp.

## 18. Final Decisions Captured by This Spec

This design intentionally adopts the following rules:

- use **MongoDB** as the NoSQL backend
- use a **Python REST API**, preferably FastAPI
- use **`Info.VPSId` as the only table identity key**
- store **`Info.Rom` and all `VPXFile` fields** as canonical global metadata
- store **`User.Rating`, `User.LastRun`, `User.StartCount`, `User.RunTime`, `User.Score`, and `VPinPlay.altvpsid`** as user-specific metadata
- ignore all other `VPinPlay` fields for this service
- ignore `Medias` for this service
- allow clients to submit a **full JSON snapshot** on startup
- let the backend compute changes and maintain current-state records

## 19. Implementation Starting Point

Recommended first implementation:

1. build FastAPI endpoints:
   - `POST /api/v1/sync` (with client validation)
   - `GET /api/v1/users/{userId}/available`
2. build read endpoints:
   - `GET /api/v1/tables/{vpsId}`
   - `GET /api/v1/users/{userId}/tables/{vpsId}`
   - `GET /api/v1/users/{userId}/tables`
3. determine collection structure and indexing based on actual usage patterns

This provides a clean and practical first version while keeping the data model flexible for future optimization.
